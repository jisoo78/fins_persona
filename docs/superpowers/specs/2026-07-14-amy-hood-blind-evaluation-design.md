# Amy Hood 페르소나 블라인드 평가 설계

## 1. 목적

`Main Master Prompt + RAG`로 구성한 Amy Hood 페르소나가 공개자료의 문장을 단순히 반복하는지, 아니면 Amy Hood의 재무 판단 스타일을 새로운 상황에도 일관되게 적용하는지 평가한다.

PoC 평가 세트는 15문항으로 제한한다.

- 과거 복원 확인 7문항: 4지선다형
- 2017~2019 GitHub 인수 홀드아웃 5문항: 4지선다형
- 가상 시나리오 3문항: 주관식

GraphRAG 비교는 이 평가 설계와 구현 범위에서 완전히 제거한다. 기존 GraphRAG 또는 RAG 포함·제외 정적 비교 산출물은 신규 평가의 입력이나 기준으로 사용하지 않는다.

## 2. 평가 원칙

### 2.1 판단 스타일 재현

평가 대상은 Amy Hood가 미래를 정확히 예측하는 능력이 아니다. 다음 판단 패턴을 얼마나 일관되게 적용하는지가 핵심이다.

- 장기 성장과 단기 수익성의 균형
- 전략적 투자와 재무 규율의 동시 유지
- 자본배분과 주주 희석 관리
- 확인된 사실과 불확실한 가정의 구분
- 인수 대상의 핵심 가치와 통합 시너지 사이의 균형

### 2.2 당시 정보만 사용

GitHub 홀드아웃 문항은 2020년 이후 알려진 성공 지표를 문제, 정답 또는 채점 근거로 사용하지 않는다. 각 문항은 해당 판단 시점까지 공개된 정보만 제시한다.

### 2.3 Gemma 4에 적합한 단순성

- 15문항을 한꺼번에 전달하지 않는다.
- 모델 호출 한 번에 문항 하나만 전달한다.
- 객관식은 번호와 1~2문장 선택 이유만 요구한다.
- 객관식 이유는 점수에 반영하지 않고 오류 분석에만 사용한다.
- 주관식은 5~8문장으로 제한한다.
- 출처 인용이나 장문의 수치 계산을 요구하지 않는다.

## 3. 블라인드 홀드아웃 경계

### 3.1 피평가 페르소나가 사용할 수 있는 자료

- 홀드아웃 기간 밖의 공개 인터뷰, 발표, 대담과 게시물
- 홀드아웃 기간 밖의 어닝콜
- 위 자료로 만든 Main Master Prompt
- 위 자료만 색인한 RAG

### 3.2 평가 제작에만 사용하는 자료

- FY2017 Q1부터 FY2019 Q4까지의 어닝콜 12개
- 2017~2019 GitHub 인수와 관련된 Microsoft·GitHub 공식 발표와 공시
- 같은 시기의 경영진 인터뷰, 공개 게시물과 당시 보도
- 평가 문항 정답과 채점 근거

이 자료는 Main Master Prompt 생성, 자료별 LLM 분석, RAG chunk 생성과 RAG 색인에서 제외한다.

### 3.3 오염 방지

각 평가 실행은 다음 값을 기록한다.

- Main Master Prompt 파일 해시
- RAG 스냅샷 ID
- RAG를 만든 원자료 ID 목록
- 질문 세트 버전
- Provider와 모델 이름

평가 실행 전 원자료 목록에 홀드아웃 source ID가 포함되어 있으면 모델을 호출하지 않고 실패한다.

객관식 문제와 정답표는 별도 파일로 분리한다. 피평가 모델에는 문제 본문과 선택지만 전달한다.

## 4. 15문항 구성

### 4.1 과거 복원 확인 7문항

과거 복원 문항은 일반적인 CFO 상식을 묻지 않는다. RAG에서 실제 사건을 찾고 당시의 우선순위와 상충관계를 복원해야 풀 수 있게 구성한다.

| ID | 사건 | 핵심 판단 영역 |
|---|---|---|
| P1 | 2014~2016 Office의 구독형 클라우드 전환 | 단기 매출 인식 변화와 반복 매출·고객 생애가치 |
| P2 | 2015~2016 휴대폰 사업 손상차손과 구조조정 | 매몰비용 인정, 실패한 투자 정리와 핵심 분야 재배치 |
| P3 | 2016 LinkedIn 인수 | 대형 인수의 자금조달, EPS 희석, 독립 운영과 장기 성장 |
| P4 | 2020 코로나19 초기 실적과 가이던스 | 불확실성 공개, 수요 변화 구분과 성급한 장기 전망 방지 |
| P5 | FY23 비용 절감과 AI 투자 병행 | 운영비 절감과 전략적 성장 투자의 선택적 보호 |
| P6 | Activision Blizzard 인수 후 통합 | 대형 자본배분, 콘텐츠 전략, 단기 마진과 통합비용 |
| P7 | FY24~FY25 AI 인프라 투자와 공급 제약 | 검증된 수요, CapEx 속도, 성장 기회와 마진 압박 |

각 문항은 4개 선택지를 가진다. 정답 외 세 선택지는 다음 오류를 서로 다르게 포함한다.

- 단기 지표 하나만 최적화한다.
- 전략적 중요성을 이유로 재무 검증을 생략한다.
- 당시 공개되지 않은 결과를 이미 아는 것처럼 판단한다.

### 4.2 GitHub 인수 홀드아웃 5문항

홀드아웃 문항은 GitHub 인수 결과를 맞히는 문제가 아니다. 인수 성공 여부가 모호했던 당시 Amy Hood의 신념을 재무적·전략적으로 압박하는 문제다.

| ID | 판단 시점 | 신념 충돌 | 정답 방향 |
|---|---|---|---|
| H1 | 2018년 6월 인수 승인 직전 | 75억 달러 가치평가와 단기 재무 가시성 부족 | 독립성·개발자 네트워크 가치를 인정하되 희석 한도와 통합 원칙을 둔 조건부 승인 |
| H2 | 인수 발표 직후 개발자 반발 | 빠른 Azure 수익화와 개발자 신뢰 | 특정 클라우드로의 강제 유도를 피하고 GitHub의 독립성·개방성을 먼저 보호 |
| H3 | FY18 Q4 자사주 매입 결정 | 전액 주식 인수의 주주 희석과 자본 유연성 | 거래 전 매입을 중단하고 종결 후 추가 매입으로 희석을 계획적으로 상쇄 |
| H4 | 2018년 10월 인수 종결 직전 | FY19·FY20 EPS 희석과 장기 영업이익 기여 | 제한된 희석을 감수하되 인수·통합 비용을 가이던스에 투명하게 반영 |
| H5 | 2019년 상반기 초기 통합 | GitHub 투자비 증가와 Microsoft 영업마진 규율 | 독립 수익을 조급하게 극대화하지 않고 생태계·기업 채택을 보되 회사 전체 마진 규율 유지 |

각 문항에는 다음 세 함정 유형을 하나씩 배치한다.

1. 단기 재무 방어형: EPS나 영업마진을 지키기 위해 필요한 장기 투자를 포기한다.
2. 무규율 전략 낙관형: 전략적으로 중요하다는 이유로 가격, 희석과 수익성 검증을 생략한다.
3. 통합 시너지 과잉형: GitHub를 Microsoft 제품에 빠르게 종속시켜 개발자 신뢰를 훼손한다.

#### H1 문항 예시

> 2018년 6월, Microsoft 이사회가 GitHub 인수를 최종 검토하고 있다. 거래 가격은 Microsoft 주식 75억 달러이며, GitHub는 2,800만 명 이상의 개발자와 150만 개 이상의 조직이 사용하는 플랫폼이다. 경영진은 FY19·FY20 비GAAP EPS 희석이 1% 미만이고 FY20 비GAAP 영업이익에는 기여할 것으로 예상하지만, GitHub 자체의 단기 재무 기여는 Microsoft 전체 규모에 비해 제한적이다. 일부 개발자는 Microsoft가 GitHub를 Azure와 자사 개발도구에 종속시킬 것을 우려하고 있다. CFO로서 이사회에 어떤 권고를 하겠는가?

1. 단기 재무 기여가 인수가격을 뒷받침하지 못하므로 GitHub가 독립적으로 EPS에 기여할 때까지 거래를 보류한다.
2. 인수를 승인하되 2년 안에 GitHub Enterprise를 Azure 중심 상품으로 전환하고 빠른 교차판매로 인수가격을 회수한다.
3. 공개한 희석 범위 안에서 인수를 승인하되 GitHub의 독립성과 플랫폼 중립성을 보호한다. 개발자 유지, 기업 채택과 Microsoft 전체 생태계 기여를 단계별로 검증하고 통합비용과 희석 영향을 지속해서 공개한다.
4. 개발자 네트워크는 대체할 수 없는 자산이므로 단기 희석이나 통합비용에 별도 한도를 두지 않고 인수를 승인한다.

정답은 3번이다.

### 4.3 가상 시나리오 3문항

가상 시나리오는 시간적 예측이 아니라 새로운 상황에 판단 스타일을 적용하는 능력을 본다.

#### S1. AI 인프라 투자와 마진 압박

Azure AI 수요가 공급을 초과한다. 추가 투자로 12개월의 공급능력을 25% 늘릴 수 있지만 선행 CapEx와 감가상각 때문에 회사 gross margin이 약 1.5%p 하락할 전망이다. 수요의 60%는 장기 계약 또는 실제 사용량으로 확인되며 나머지는 고객 전망에 의존한다.

투자를 승인·축소·단계화할지 결정하고 이유, 승인 조건과 확인할 지표를 설명하게 한다.

#### S2. 개발자 AI 플랫폼 인수

Microsoft가 빠르게 성장하는 개발자 AI 플랫폼을 120억 달러에 인수할 기회를 얻었다. 현재 매출과 영업이익은 작지만 개발자 네트워크와 데이터 자산의 전략적 가치는 높다. Microsoft 제품에 빠르게 통합하면 단기 매출 시너지가 예상되지만 플랫폼 중립성을 잃으면 개발자 이탈 가능성이 있다. 거래는 주식과 현금을 함께 사용하며 첫 2년간 EPS를 약 1.5% 희석할 전망이다.

인수 여부, 거래 조건, 독립 운영 범위와 재무적 레드라인을 제시하게 한다.

#### S3. 경기 둔화기의 선택적 비용 절감

기업 IT 지출 둔화로 신규 계약과 사용량 증가율이 낮아지고 있다. 다음 회계연도 운영비를 8% 줄여야 하지만 AI 인프라, 보안과 핵심 클라우드 연구개발은 장기 성장에 중요하다. 성숙 제품 마케팅, 중복 관리조직과 저성과 지역 영업조직에도 비용이 배정되어 있다.

먼저 줄일 비용과 보호할 투자를 결정하고 투자자 가이던스에서 확정된 사실과 불확실한 가정을 구분하게 한다.

## 5. 응답과 채점 계약

### 5.1 객관식 12문항

모델 응답 형식은 다음과 같다.

```json
{
  "choice": 3,
  "reason": "장기 전략 가치를 인정하되 독립성과 희석 한도를 함께 관리해야 합니다."
}
```

- `choice`가 정답이면 1점, 아니면 0점이다.
- `reason`은 1~2문장으로 제한한다.
- `reason`은 정량 점수에 반영하지 않는다.
- `reason`은 우연히 맞힌 답, 과도한 사후확신과 프롬프트 오류를 사람이 진단하는 데 사용한다.

### 5.2 주관식 3문항

각 답변을 다음 네 항목에서 0~2점으로 채점한다.

| 항목 | 판단 기준 |
|---|---|
| 결론 | 명확한 결정과 필요한 조건을 제시했는가 |
| 근거 | 시나리오에 주어진 핵심 사실을 판단에 사용했는가 |
| 상충관계 | 성장, 마진, 위험과 기회비용을 함께 비교했는가 |
| 페르소나 일관성 | 장기 성장 투자와 재무 규율을 동시에 유지하는가 |

문항당 8점, 세 문항 합계 24점이다. 문장 표현의 세련됨보다 결정 논리가 주어진 정보와 연결되는지를 우선한다.

채점 결과 형식은 다음과 같다.

```json
{
  "score": 6,
  "decision": 2,
  "reasoning": 2,
  "tradeoff": 1,
  "persona_consistency": 1,
  "summary": "투자 방향은 명확하지만 수요 검증 조건과 중단 기준이 부족하다."
}
```

초기 PoC에서는 Codex가 같은 계약으로 채점할 수 있다. 이후 API 채점 모델로 교체해도 저장 형식은 바꾸지 않는다.

### 5.3 블라인드 채점

주관식 채점 입력에는 다음 정보만 포함한다.

- 문항
- 문항별 채점 기준
- 익명화된 답변

생성 Provider와 모델, 이전 평가 점수와 기대 순위는 채점 입력에서 제외한다. Provider와 모델 정보는 실행 기록에 보존하고 채점 완료 후 사용자 비교 화면에 표시한다.

## 6. 모델 입력 조립

| KPI | 모델 입력 |
|---|---|
| 과거 복원 7문항 | Main Master Prompt + 검색된 소수 RAG chunk + 문항 1개 |
| GitHub 홀드아웃 5문항 | Main Master Prompt + 문항 1개 |
| 가상 시나리오 3문항 | Main Master Prompt + 시나리오 1개 |

과거 복원에서도 전체 chunk를 다시 주입하지 않는다. 질문별 검색 결과만 사용한다. 홀드아웃과 가상 시나리오에는 RAG 검색 결과를 붙이지 않는다.

## 7. 파일 계약

PoC는 데이터베이스를 추가하지 않고 파일 기반으로 유지한다.

```text
evaluation/amy_hood_eval_questions.json
evaluation/amy_hood_eval_answer_key.json
evaluation/amy_hood_eval_question_reviews.json
evaluation/runs/{run_id}.json
```

### 7.1 질문 파일

피평가 모델에 전달할 문항, 선택지, KPI와 버전만 저장한다. 정답이나 함정 설명은 포함하지 않는다.

### 7.2 정답 파일

객관식 정답, 정답 의도, 오답별 함정 의도와 주관식 채점 기준을 저장한다. 평가 실행 서버만 읽는다.

### 7.3 검토 파일

문항별로 다음 값을 저장한다.

- `question_id`
- `status`: `unreviewed`, `approved`, `revision_required`
- `revision_note`
- `reviewed_at`

UI는 질문 원본을 직접 수정하지 않는다. Codex가 수정 지시를 반영한 뒤 질문 세트 버전을 올리고 사용자가 다시 검토한다.

### 7.4 실행 기록

```json
{
  "run_id": "uuid",
  "status": "running",
  "provider": "local",
  "model": "gemma-4",
  "prompt_hash": "sha256",
  "rag_snapshot_id": "snapshot-id",
  "question_set_version": "1.0.0",
  "answers": [],
  "scores": {},
  "started_at": "ISO-8601",
  "completed_at": null
}
```

문항 하나가 끝날 때마다 실행 기록을 원자적으로 갱신한다. 중간 실패 시 완료한 응답은 보존하되 실행 상태를 `incomplete`로 두고 완료된 비교 결과에는 포함하지 않는다. 재시도는 실패한 문항부터 재개한다.

## 8. Web UI

### 8.1 평가 문항 검토

사이드바에 `평가 문항 검토` 메뉴를 추가한다.

상단에는 전체 문항 수, KPI별 문항 수와 `승인`, `수정 필요`, `미검토` 개수를 표시한다. KPI와 검토 상태로 필터링할 수 있다.

각 문항 카드는 다음 내용을 보여준다.

- 문항 ID와 KPI
- 문제 본문
- 객관식 선택지 4개
- 정답과 정답 의도
- 오답별 함정 의도
- 주관식 채점 기준
- `승인`과 `수정 필요` 선택
- 수정 지시 메모

검토 상태와 메모만 별도 검토 파일에 저장한다.

### 8.2 평가 비교

기존 `평가 비교` 메뉴는 유지한다. 현재의 정적 RAG·GraphRAG 근거 비교표는 제거하고 실제 평가 실행과 이력 비교 화면으로 교체한다.

실행 전 다음 값을 표시한다.

- Provider와 모델
- Main Master Prompt 버전
- RAG 스냅샷
- 평가 문항 버전
- 평가 실행 버튼

실행 중에는 현재 문항, 완료 문항 수, 실패 문항 수와 중단·재개 상태를 표시한다.

실행 완료 후 다음 KPI를 표시한다.

- 과거 복원 정확도: 7점 만점
- GitHub 홀드아웃 정확도: 5점 만점
- 가상 시나리오 점수: 24점 만점
- 전체 실행시간
- 실패와 재시도 문항 수

이력에서 두 실행을 선택해 Gemma 4와 GPT-5 mini, Prompt 버전 또는 RAG 스냅샷 차이를 비교할 수 있다. 문항별 상세에는 모델 응답, 정답 여부, 주관식 세부 점수와 실행 메타데이터를 표시한다.

## 9. 서버 동작

평가 실행은 장시간 HTTP 요청 하나로 묶지 않는다.

1. 실행 생성 요청은 `run_id`와 `running` 상태를 즉시 반환한다.
2. 서버는 문항을 순차적으로 실행하고 매 문항 뒤 기록을 갱신한다.
3. UI는 실행 상태를 조회해 진행률을 갱신한다.
4. 서버 재시작이나 모델 오류가 발생하면 실행을 `incomplete`로 표시한다.
5. 사용자는 같은 설정으로 미완료 문항부터 재개할 수 있다.

질문 검토 저장과 평가 실행 기록은 임시 파일에 먼저 쓴 뒤 rename하는 원자적 쓰기를 사용한다.

## 10. 오류 처리

- 홀드아웃 오염: 모델 호출 전 실행을 거부한다.
- 질문과 정답 ID 불일치: 평가 실행을 시작하지 않는다.
- 잘못된 객관식 응답: 해당 문항을 실패로 기록하고 한 번만 형식 재요청한다.
- 모델 호출 실패: 완료 응답을 보존하고 실행을 `incomplete`로 둔다.
- 주관식 채점 실패: 생성 답변은 보존하고 채점 상태만 `pending`으로 둔다.
- 검토 저장 실패: 기존 검토 파일을 유지하고 UI에 명확한 오류를 표시한다.
- 미승인 질문 세트: 기본 실행을 막고 검토 화면으로 이동할 수 있게 한다.

## 11. TDD 검증 전략

새로 만들거나 크게 수정하는 테스트 파일 상단에는 프로젝트 `AGENTS.md` 형식의 Test Plan 주석을 먼저 작성한다.

### Happy Path 1개

- 승인된 15문항을 Gemma 4 모의 모델로 순차 실행해 객관식과 주관식 결과를 저장하고 KPI 요약을 생성한다.

### Edge Cases 정확히 3개

1. 객관식 모델 응답에 설명 문장이 포함되어도 선택 번호와 이유를 정상적으로 분리한다.
2. 실행이 중단된 뒤 재개하면 완료한 문항을 다시 호출하지 않고 미완료 문항부터 진행한다.
3. 검토 메모가 비어 있는 승인 상태와 한국어 수정 메모가 있는 `revision_required` 상태를 각각 보존한다.

### Failure Paths

- RAG 스냅샷에 홀드아웃 source ID가 있으면 모델을 한 번도 호출하지 않는다.
- 질문 파일과 정답 파일의 ID가 다르면 실행 기록을 생성하지 않는다.
- 모델 또는 채점 API 실패 시 손상되거나 완료로 표시된 결과를 만들지 않는다.
- 검토 파일 저장 실패 시 기존 파일을 덮어쓰거나 일부만 변경하지 않는다.

테스트를 먼저 작성하고 실패를 확인한 뒤 최소 구현을 추가한다. 리팩터링 중에는 기대 동작이 바뀌지 않는 한 기존 테스트를 약화하거나 삭제하지 않는다.

## 12. PoC 완료 기준

- 7개 과거 복원, 5개 GitHub 홀드아웃, 3개 가상 시나리오 문항이 버전 관리된다.
- 객관식 문제와 정답표가 분리된다.
- GitHub 홀드아웃 자료가 Main Master Prompt와 RAG에 포함되지 않는다.
- `평가 문항 검토`에서 15문항을 한눈에 확인하고 승인 또는 수정 지시를 저장할 수 있다.
- `평가 비교`에서 실행, 진행률 확인, 실패 후 재개와 이력 비교가 가능하다.
- Gemma 4와 GPT-5 mini가 동일한 질문 세트와 입력 계약을 사용한다.
- 생성 모델 정보를 숨긴 상태로 주관식 채점이 가능하다.
- 실행 기록에서 Prompt, RAG, 질문 세트와 모델 버전을 재현할 수 있다.
- GraphRAG 평가 입력과 UI 의존성이 남아 있지 않다.

## 13. 제외 범위

- GraphRAG 구축 또는 성능 비교

- 실시간 외부 자료 검색을 평가 중 허용하는 기능

- 다중 사용자 권한과 외부 평가 서비스
