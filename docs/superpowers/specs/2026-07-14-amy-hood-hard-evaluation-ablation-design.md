# Amy Hood 고난도 문항과 3조건 절제 실험 설계

## 1. 목적

현재 Amy Hood 평가의 객관식 12문항은 파이프라인 정상 작동을 확인하는 데는 유효하지만 페르소나의 판단 스타일을 변별하기에는 쉽다. 모든 정답이 가장 긴 선택지이고, 오답 다수가 극단적이어서 일반적인 CFO 추론만으로도 제거할 수 있다. 실제 Gemma 4 실행에서도 RAG가 없는 GitHub 홀드아웃 5문항을 모두 맞혔다.

이번 변경은 다음 두 질문에 답할 수 있는 평가를 만든다.

1. 검색된 공개자료가 과거 판단 복원 성능을 실제로 높이는가?
2. Amy Hood Main Prompt가 일반 CFO Prompt보다 Amy Hood에 가까운 판단을 유도하는가?

평가 규모와 KPI는 유지한다.

- 과거 복원 확인 7문항: 4지선다형
- 2017~2019 GitHub 인수 홀드아웃 5문항: 4지선다형
- 가상 시나리오 3문항: 주관식

GraphRAG는 범위에 포함하지 않는다.

## 2. 성공 기준

### 2.1 문항 품질

- 객관식 네 선택지는 모두 현실적인 CFO 대안이어야 한다.
- 정답은 단순히 가장 온건하거나 가장 긴 선택지가 아니어야 한다.
- 오답은 사실 오류가 아니라 판단 순서, 증거 가중치, 적용 시점 또는 재무 레드라인이 미묘하게 다른 근접 오답이어야 한다.
- 정답 길이는 같은 문항의 오답 평균 길이와 비교해 `±10%` 이내여야 한다.
- 정답이 유일한 최장 선택지인 문항은 전체 12개 중 최대 3개다.
- 정답 위치 1~4는 각 위치에 2~4개씩 배치한다.
- `모든`, `무조건`, `영구적으로`, `한도를 두지 않고`, `전액을 한 분기에`처럼 선택지만 보고 제거할 수 있는 절대 표현을 함정의 핵심으로 사용하지 않는다.

### 2.2 실험 품질

- 세 실험군은 동일한 Gemma 4 프로세스, 문항 버전, 생성 설정과 실행 순서를 사용한다.
- 실험군 간 차이는 Main Prompt와 RAG 사용 여부뿐이어야 한다.
- 세 실행은 하나의 `experimentGroupId`로 연결한다.
- 기존 단일 실행과 과거 실행 파일은 계속 읽을 수 있어야 한다.
- 평가 비교 UI에서 세 실험군의 과거 복원, 홀드아웃, 주관식 점수와 lift를 확인할 수 있어야 한다.

### 2.3 해석 기준

이 PoC는 사전에 합격 점수를 고정하지 않는다. 먼저 실제 차이를 측정한다.

- `RAG lift = persona_rag.pastMemory - persona_no_rag.pastMemory`
- `Persona lift = persona_no_rag.githubHoldout - generic_cfo_no_rag.githubHoldout`
- 가상 시나리오는 실험군을 숨긴 동일 채점 계약으로 비교한다.

일반 CFO 통제군이 객관식 대부분을 맞히거나 두 lift가 모두 0에 가까우면, 페르소나 성공으로 해석하지 않고 문항 또는 Main Prompt의 변별력 부족으로 기록한다.

## 3. 문항 재작성 원칙

### 3.1 근접 오답의 다섯 축

각 객관식 문항은 정답과 세 오답을 다음 축 중 서로 다른 축에서 갈라지게 한다.

1. **승인 순서**: 즉시 승인, 조건부 승인, 단계적 승인, 추가 증거 후 승인 중 무엇을 먼저 택하는가.
2. **지표 우선순위**: 계약, 실제 사용량, 매출 전환, 마진, 현금흐름, 참여도 중 어떤 지표를 선행 신호로 보는가.
3. **레드라인 적용 시점**: 거래 승인 전, 통합 초기, 중기 성과 검토 또는 포트폴리오 재평가 시점 중 어디에 재무 한도를 두는가.
4. **독립성과 수익화 순서**: 플랫폼 신뢰를 먼저 보호할지, 공동판매를 먼저 확대할지, 제품 통합을 먼저 진행할지의 차이다.
5. **증거의 확실성**: 계약된 수요, 실제 사용량, 고객 전망과 전략적 기대를 서로 다른 확률과 시간축으로 해석하는 차이다.

### 3.2 선택지 공통 구조

네 선택지는 모두 다음 요소를 포함한다.

- 명확한 행동 또는 권고
- 그 행동이 보호하려는 가치
- 감수하는 비용 또는 위험
- 다음 검토 조건이나 지표

따라서 “투자를 모두 중단한다”와 같은 단문 오답 대신, 다른 우선순위를 가진 실행 가능한 대안을 제시한다.

### 3.3 과거 복원 P1~P7

과거 복원 문항의 정답은 검색 원문에 나타난 사건별 판단 순서를 반영한다. 오답도 같은 사건의 실제 수치와 전략적 맥락을 사용하지만 다음 중 하나를 다르게 둔다.

- 후행지표를 선행지표보다 먼저 요구한다.
- 포트폴리오 가치보다 독립 사업 손익을 먼저 최적화한다.
- 검증된 수요와 전망 수요에 같은 가중치를 준다.
- 필요한 투자를 보호하지만 운영 레버리지 이정표를 늦게 적용한다.

질문 본문은 정답 선택지의 판단 순서를 그대로 암시하지 않는다. `retrievalQuery`는 문항 본문과 분리해 유지한다.

### 3.4 GitHub 홀드아웃 H1~H5

네 선택지 모두 GitHub의 개발자 네트워크 가치, 플랫폼 신뢰, EPS 희석과 Microsoft 생태계 기여를 인정한다. 차이는 다음 순서에서 만든다.

- 독립 운영을 거래 조건으로 먼저 고정하는가, 통합 과정에서 추후 조정하는가.
- 개발자 유지와 기업 채택을 선행지표로 보는가, Azure 교차판매와 독립 손익을 먼저 보는가.
- 희석 한도를 승인 조건으로 두는가, 종결 후 자사주 매입으로 우선 상쇄하는가.
- 알려진 거래비용을 즉시 가이던스에 반영하는가, 구매회계 확정 시점까지 범위를 넓게 유지하는가.

평가 문항과 정답은 2020년 이후 GitHub의 성공 결과를 사용하지 않는다.

### 3.5 가상 시나리오 S1~S3

주관식 3문항은 현재 네 차원 채점 계약을 유지한다.

- 결정: 0~2점
- 근거: 0~2점
- 상충관계: 0~2점
- 페르소나 일관성: 0~2점

질문에서 `승인·축소·단계화`처럼 기대 행동을 열거하지 않는다. 모델이 스스로 행동과 순서를 정하도록 시나리오 사실, 제약과 불확실성만 제시한다.

## 4. 세 실험군

`EvaluationExperimentArm`은 다음 세 값으로 고정한다.

| 실험군 | System Prompt | P1~P7 RAG | H1~H5 RAG | S1~S3 RAG |
|---|---|---:|---:|---:|
| `persona_rag` | 실행에 고정된 활성 Amy Hood Prompt | 사용 | 미사용 | 미사용 |
| `persona_no_rag` | 같은 활성 Amy Hood Prompt | 미사용 | 미사용 | 미사용 |
| `generic_cfo_no_rag` | 고정된 일반 CFO 통제 Prompt | 미사용 | 미사용 | 미사용 |

세 실행은 `local` Provider만 사용하고 8080의 동일 Gemma 4 서버에서 순차 실행한다. `--parallel 1` 런타임과 충돌하지 않도록 동시에 호출하지 않는다.

### 4.1 일반 CFO 통제 Prompt

일반 CFO Prompt는 다음 조건을 만족하는 별도 Markdown 자산으로 둔다.

- Amy Hood, Microsoft 또는 특정 과거 사건을 언급하지 않는다.
- 시나리오에 주어진 사실만 사용한다.
- 재무와 전략의 장단점을 비교하고 명확한 결론을 내린다.
- 답변 언어와 객관식 JSON 형식은 Amy Hood 실험군과 같다.
- Amy Hood Main Prompt의 고유 판단 규칙이나 공개자료 내용을 복제하지 않는다.

실행 기록에는 통제 Prompt의 SHA-256을 저장한다.

## 5. 실제 system role 분리

현재 평가기는 `[SYSTEM PERSONA]`라는 텍스트 태그를 하나의 문자열 안에 넣는다. 변경 후 모델 입력은 역할을 분리한다.

```text
system:
  <Amy Hood 활성 Prompt 또는 일반 CFO 통제 Prompt>

user:
  <RAG 증거 블록, 질문, 선택지, 출력 계약>
```

`ModelClient`는 기존 문자열 `invoke(prompt)` 사용처를 깨지 않으면서 평가기가 역할 메시지를 전달할 수 있는 명시적 채팅 입력 인터페이스를 제공한다. 분석 파이프라인의 기존 문자열 호출은 그대로 동작한다.

Main Prompt는 RAG 원문과 질문보다 높은 역할 우선순위를 갖는다. RAG 내부 명령문이나 역할 변경 문장은 증거로만 취급한다.

## 6. RAG 증거와 메타데이터

`persona_rag`의 P1~P7만 검색 결과를 받는다. 현재 PoC의 결정적 키워드 top-1 검색은 이번 실험에서 유지해 문항 난이도 변경과 검색기 변경을 섞지 않는다.

모델에 전달하는 증거 형식은 다음과 같다.

```text
[RAG EVIDENCE]
source_id: earnings_fy2024_q4
chunk_id: earnings_fy2024_q4:0:5e5157c4b7f2
chunk_index: 0
block_ids: turn-1, turn-2, ...

<speaker label을 포함한 원문 chunk.text>
```

메타데이터는 현재 manifest에 이미 존재하는 값만 사용한다. 날짜, 문서 유형, 화자 또는 출처 URL을 추정하거나 새로 생성하지 않는다. H1~H5와 S1~S3에는 메타데이터와 원문을 모두 넣지 않는다.

사용자 응답에 출처 연결을 강제하지 않는다. 메타데이터는 모델의 내부 맥락 구분과 실행 재현에만 사용한다.

## 7. 실행 및 저장 계약

### 7.1 실행 타입 확장

기존 `EvaluationRun`에 다음 선택 필드를 추가한다.

```ts
type EvaluationExperimentArm =
  | 'persona_rag'
  | 'persona_no_rag'
  | 'generic_cfo_no_rag';

type EvaluationRun = {
  // 기존 필드
  experimentGroupId?: string;
  experimentArm?: EvaluationExperimentArm;
};
```

기존 실행 파일은 두 필드가 없어도 정상 로드한다. 일반 단일 실행은 명시하지 않으면 `persona_rag` 동작을 사용한다.

### 7.2 실험 생성 API

`POST /api/evaluation/experiments`는 다음 계약을 사용한다.

```json
{
  "provider": "local"
}
```

서버는 UUID 실험 그룹을 만들고 세 실행을 고정 순서로 생성한다.

1. `persona_rag`
2. `persona_no_rag`
3. `generic_cfo_no_rag`

응답은 `202`와 그룹 ID, 세 run ID를 반환한다. 서버는 백그라운드에서 세 실행을 순차 처리한다. 한 실행이 실패해도 완료·실패 상태를 보존하고 다음 실험군을 실행한다.

지원하지 않는 Provider, 알 수 없는 실험군, 같은 그룹 안의 중복 실험군은 모델 호출 전에 거부한다.

### 7.3 질문과 검토 버전

- 질문, 정답표와 검토 파일 버전을 `2.0.0`으로 함께 올린다.
- 15개 검토 항목은 사용자의 이번 명시적 승인에 따라 모두 `approved`로 저장한다.
- `reviewedAt`에는 실제 저장 시각을 기록한다.
- 기존 `1.0.0` 실행 기록은 변경하지 않는다.

## 8. Web UI

평가 실행 화면에 `3조건 실험 실행` 버튼을 추가한다. 버튼은 실행 중 중복 제출을 막고, 생성된 `experimentGroupId`와 세 run ID를 표시한다.

실행 목록과 리포트에는 다음 라벨을 표시한다.

- `Amy Hood + RAG`
- `Amy Hood / RAG 없음`
- `일반 CFO / RAG 없음`

실험 그룹 카드는 세 실행의 상태를 함께 표시한다. 세 실행이 완료되면 다음 표를 보여준다.

| 조건 | 과거 복원 /7 | GitHub 홀드아웃 /5 | 주관식 /24 |
|---|---:|---:|---:|
| Amy Hood + RAG | 값 | 값 | 값 또는 채점 대기 |
| Amy Hood / RAG 없음 | 값 | 값 | 값 또는 채점 대기 |
| 일반 CFO / RAG 없음 | 값 | 값 | 값 또는 채점 대기 |

표 아래에 RAG lift와 Persona lift를 표시한다. 주관식이 채점 전이면 객관식 lift는 즉시 계산하고 주관식은 `채점 대기`로 둔다.

기존 두 실행 비교 기능, 단일 실행 리포트, 실행 ID 복사와 외부 주관식 채점은 유지한다.

## 9. 테스트 설계

새로 만들거나 크게 수정하는 테스트 파일 상단에는 AGENTS.md 형식의 Test Plan 주석을 둔다.

### 9.1 Happy Path

- 동일한 문항 버전, 모델 설정과 실험 그룹 ID를 가진 세 실행을 순차 생성한다.
- 각 실험군이 정확한 System Prompt와 RAG 정책을 사용한다.
- UI가 세 실행 점수와 두 lift를 표시한다.

### 9.2 Edge Cases — 기본 세 가지

1. 실험 필드가 없는 기존 실행을 legacy 단일 실행으로 정상 표시한다.
2. `persona_no_rag`와 `generic_cfo_no_rag`는 P/H/S 모든 문항에서 빈 증거 배열을 사용한다.
3. 선택지 순서와 정답 위치가 달라도 정답 및 `trapIntents` 연결을 유지한다.

문항 데이터에는 별도의 품질 검증을 적용한다.

- 정답과 오답 평균 길이 차이 `±10%`
- 정답이 유일한 최장 선택지인 문항 최대 3개
- 정답 위치별 2~4개
- 질문/정답/검토 파일 ID와 버전 일치

### 9.3 Failure Paths

- `openai` 등 `local` 이외 Provider로 3조건 실험을 요청하면 모델 호출 없이 실패한다.
- 알 수 없는 실험군 또는 한 그룹의 중복 실험군을 거부한다.
- RAG 없음 조건에 청크가 전달되면 프롬프트 생성 전에 실패한다.
- 실험 중 한 실행이 실패해도 이미 완료된 실행 파일을 손상시키지 않고 다음 실험군을 진행한다.
