# Amy Hood Contrastive Reflection Gate Design

## 1. 목적

Amy Hood Decision Advisor가 서로 다른 사건을 단순히 나열하지 않고, 동일한 경영 판단 축에서 조건 변화가 실제 행동 변화를 일으킨 사례만 대조 성찰로 승인하도록 강화한다. 목표는 현재 승인 사건 범위 안에서 근거가 있는 성찰과 조건부 정책을 최소 1개 도출하고, PoC용 구조화 메모리 릴리스를 활성화하는 것이다.

이 변경은 판단 정책의 품질 게이트에만 집중한다. 사건 수집, Evaluation v3 문항, RAG 검색 방식, Main Prompt는 변경하지 않는다.

## 2. 현재 문제

현재 `validateReflectionMemory`는 다음만 확인한다.

- 지원·대조 사건 집합이 비어 있지 않음
- 두 집합이 서로 겹치지 않음
- 사건과 증거 ID가 현재 입력 그래프에 존재함
- 지원 사건의 도메인이 성찰 도메인과 일치함
- 홀드아웃 증거가 포함되지 않음

따라서 다음 의미 오류를 차단하지 못한다.

- 서로 다른 판단 질문을 다루는 사건을 대조로 연결
- 조건과 행동이 같은 보완 사례를 반례로 표시
- 사건에 존재하지 않는 전략적 이유를 인용 근거 없이 추론
- 한쪽 조건 또는 행동에 대응하는 원문 증거가 없음

2026-07-20 실제 Gemma 4 실행에서 형식 검증을 통과한 성찰 3건은 Codex 의미 검토를 통과하지 못했다.

| 성찰 | 자동 결과 | 의미 검토 결과 | 핵심 결함 |
|---|---|---|---|
| 인수 대 파트너십 | 통과 | 거절 | 인용 스팬이 통제 필요성·전략적 도달 범위·통합 역량을 뒷받침하지 않음 |
| 부채 대 자본조달 | 통과 | 거절 | Activision 사건에 비교 가능한 조달 선택 근거가 없음 |
| 인력 재배치 대 OpenAI 투자 | 통과 | 거절 | 두 사건은 반대가 아니라 성장 분야 집중이라는 같은 방향의 지원 사례임 |

## 3. 공식 정의

### 3.1 판단 축

판단 축은 산업이나 사건 분류가 아니라 다음의 조합이다.

> 판단 대상 + 비교 가능한 선택지 + 선택을 전환하는 관찰 가능 조건

예:

- 판단 대상: 전략적 자원 배분
- 선택지: 확대 / 유지 / 축소 또는 재배치
- 전환 조건: 고객 수요, 장기 성장 기회, 자원 생산성

### 3.2 적격 대조

다음 조건을 모두 만족할 때만 적격 대조로 인정한다.

1. 지원 사건과 대조 사건이 같은 판단 대상을 다룬다.
2. 양쪽 행동이 같은 선택지 집합 안에서 비교 가능하다.
3. 관찰 가능한 조건 차이가 존재한다.
4. 조건 차이가 실제 행동 차이를 설명한다.
5. 지원·대조 양쪽의 조건과 행동에 원문 증거가 연결된다.

도메인은 달라도 같은 판단 축이면 허용한다. 그러나 도메인이 다르다는 사실 자체는 대조 근거가 될 수 없다.

## 4. 구조화 출력 계약

Gemma 4는 기존 성찰 필드와 함께 다음 구조를 JSON으로 출력한다.

```json
{
  "decisionAxis": {
    "decisionObject": "strategic_resource_allocation",
    "decisionQuestion": "When should resources be expanded versus reduced or reallocated?",
    "choiceSet": ["expand", "maintain", "reduce_or_reallocate"],
    "gatingVariables": [
      "observable_customer_demand",
      "long_term_growth_opportunity",
      "resource_productivity"
    ]
  },
  "supportPattern": {
    "eventIds": ["event-support"],
    "conditions": ["substantial growth opportunity remains observable"],
    "action": "expand",
    "evidenceIds": ["span-support"]
  },
  "contrastPattern": {
    "eventIds": ["event-contrast"],
    "conditions": ["demand or productivity weakens"],
    "action": "reduce_or_reallocate",
    "evidenceIds": ["span-contrast"]
  },
  "conditionDelta": "growth evidence changes from substantial to insufficient",
  "actionDelta": "expand changes to reduce_or_reallocate"
}
```

필드의 목적은 모델의 설명을 늘리는 것이 아니라, 코드와 검토자가 동일한 검사표로 결과를 판정하게 하는 것이다.

## 5. 검증 아키텍처

### 5.1 Gemma 제안

성찰 프롬프트에 적격 대조의 정의와 금지 사례를 추가한다. Gemma는 공급된 사건·증거만 사용하고 구조화 계약에 맞는 JSON을 반환한다. 기존과 동일하게 잘못된 JSON에는 1회만 복구 재시도를 허용한다.

### 5.2 결정적 코드 검증

`validateReflectionMemory`는 다음을 추가로 검사한다.

- 판단 축 필드와 패턴 필드가 비어 있지 않음
- `choiceSet`에 지원·대조 행동이 모두 포함됨
- 지원·대조 행동이 서로 다름
- `conditionDelta`와 `actionDelta`가 비어 있지 않음
- 각 패턴의 사건 ID가 해당 지원·대조 집합과 정확히 대응함
- 각 패턴에 최소 하나의 조건과 증거 ID가 있음
- 패턴 증거가 해당 패턴 사건에 실제로 속함
- 패턴 증거의 합집합이 최상위 `evidenceIds`와 일치함
- 홀드아웃 및 post-outcome 증거가 없음

코드는 자연어의 진실성을 단독 판정하지 않는다. 구조와 참조 무결성을 판정한다.

### 5.3 Codex 의미 검토

자동 검증 통과 결과도 `review_required`로 남긴다. Codex는 다음을 확인한 뒤에만 승인한다.

- 동일 판단 축이 억지로 일반화되지 않았는가
- 양쪽 행동이 실제로 다르고 중요성이 있는가
- 조건 차이가 행동 차이의 합리적 원인인가
- 인용문이 조건과 행동을 직접 또는 제한적으로 뒷받침하는가
- 사후 성공 결과나 알려지지 않은 내부 동기를 사용하지 않았는가

## 6. 데이터 흐름

1. 승인 사건 5개와 검증 증거를 봉인 입력 그래프로 로드한다.
2. Gemma 4가 강화된 JSON 계약으로 성찰을 제안한다.
3. 결정적 검증기가 구조·ID·근거 연결·홀드아웃을 검사한다.
4. 실패 제안은 승인 경로에서 제외하고 정확한 오류를 기록한다.
5. 자동 통과 제안은 Codex가 원문과 사건 카드를 검토한다.
6. 승인 성찰만 Gemma 정책 유도 입력으로 사용한다.
7. 정책의 조건·우선순위·예외·반전 신호를 다시 Codex가 검토한다.
8. 승인 정책이 최소 1개일 때만 불변 메모리 릴리스를 만들고 활성화한다.

## 7. 오류 처리와 PoC 중단 기준

- JSON 파싱 실패: 기존과 같이 정확히 1회 복구 재시도 후 모델 런을 실패로 저장한다.
- 구조 또는 근거 연결 실패: 제안을 저장하되 승인 대상에서 제외하고 오류를 게이트 보고서에 기록한다.
- Codex 의미 검토 실패: 승인하지 않고 근거 부족 사유를 남긴다.
- Gemma 생성은 최대 2회 실행한다. 같은 입력으로 무한 반복하지 않는다.
- 두 실행 후 승인 성찰이 0개이면 데이터 부족으로 중단한다.
- 승인 정책이 0개이면 메모리 릴리스와 Evaluation v3 RAG 실험을 실행하지 않는다.
- 기존 활성 릴리스가 있다면 실패 과정에서 변경하지 않는다.

## 8. 테스트 설계

AGENTS.md 규칙에 따라 변경 테스트에는 다음을 포함한다.

### Happy Path

- 도메인은 다르지만 같은 자원 배분 판단 축이고, 조건과 행동이 실제로 다른 성찰이 통과한다.

### Edge Cases

1. 같은 도메인이며 동일 판단 축인 적격 대조가 통과한다.
2. 선택지는 다르지만 동의어로 정규화 가능한 행동은 중복 행동으로 차단한다.
3. 지원·대조 패턴의 증거 순서가 달라도 같은 집합이면 통과한다.

### Failure Paths

- 조건은 다르지만 행동이 같은 보완 사례를 대조로 제출
- 행동은 다르지만 동일 판단 축이 아님
- 패턴 사건과 증거 소유 사건이 불일치
- 한쪽 조건 또는 증거가 비어 있음
- 최상위 증거와 패턴 증거 합집합이 불일치
- 홀드아웃 또는 post-outcome 증거 포함
- 두 번의 모델 응답이 모두 계약 위반

## 9. 완료 기준

- 강화된 성찰 테스트와 기존 전체 회귀 테스트가 통과한다.
- 기존 3개 부적격 성찰이 새 규칙 또는 Codex 의미 검토에서 승인되지 않는다.
- 새 계약으로 생성한 성찰 중 최소 1개가 자동 검증과 Codex 검토를 모두 통과한다.
- 승인된 성찰에서 검증 가능한 정책이 최소 1개 생성된다.
- 불변 메모리 릴리스가 생성·검증·활성화된다.
- 홀드아웃 누출은 0건이다.
- 이후 Evaluation v3 4개 실험군 1회 실행이 가능하다.

## 10. 비범위

- 새로운 사건 또는 웹 자료 수집
- 별도 LLM 심판 도입
- 임베딩 검색 또는 BGE-M3 도입
- Main Prompt 재작성
- Evaluation v3 문항 변경
- 페르소나 복제 성능에 대한 최종 주장
