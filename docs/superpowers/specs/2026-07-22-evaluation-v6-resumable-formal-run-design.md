# Evaluation v6 Resumable Formal Run Design

## 목표

Evaluation v6의 30개 시나리오를 3개 실험군에서 5회 실행해 450개 응답을 생성하고, 450개 개별 정체성 채점과 225개 행동 전환 채점을 중단 후 재개 가능한 정식 파이프라인으로 완성한다.

## 고정 서비스 구성

- 응답 생성 Gemma 4: `http://127.0.0.1:8080/v1`
- BGE-M3 임베딩: `http://127.0.0.1:8081/v1`
- 채점 Gemma 4: `http://127.0.0.1:8082/v1`
- 세 서비스의 실제 모델 ID는 각 `/models` 응답으로 발견하고 실행 기록에 고정한다.

## 권장 구조

기존 Evaluation v6 실행기와 채점기를 유지하면서 정식 실행 오케스트레이터를 추가한다. 오케스트레이터는 생성, 개별 채점, 쌍 채점, 보고서 작성을 순서대로 호출하되 각 단계가 이미 완료되었다면 검증 후 건너뛴다.

개별 채점 초안은 실험 그룹과 반복 번호를 함께 경로에 포함한다. 각 반복은 90개 패킷을 독립적으로 저장하며, 완료된 반복 결과를 병합해 정확히 450개의 최종 개별 점수 배치를 활성화한다. 따라서 2회차가 1회차 초안과 충돌하지 않고, 프로세스가 종료되어도 마지막 완료 패킷 다음부터 재개할 수 있다.

## 실행 흐름

1. 동결된 v6 번들, 활성 프롬프트, 메모리 릴리스, 검색 인덱스, Judge calibration을 검증한다.
2. 5회 × 3개 실험군의 15개 run을 생성하거나 기존 그룹을 불러온다.
3. 각 run은 이미 완료된 문항을 건너뛰며 총 450개 응답을 완성한다.
4. 반복 1~5마다 90개 blind packet을 채점하고 반복별 초안에 원자적으로 저장한다.
5. 다섯 반복이 모두 완료되면 packet ID와 hash를 검증해 450개 점수를 하나의 활성 배치로 병합한다.
6. 225개 initial/changed pair를 채점하며 완료된 pair는 건너뛴다.
7. 450개 개별 점수와 225개 pair 점수가 모두 활성화된 경우에만 HTML 보고서를 생성한다.

## 중단 및 재개 규칙

- 응답 생성은 기존 run JSON의 `complete` 답변을 재사용한다.
- 개별 채점은 반복별 초안에서 완료된 packet ID를 재사용한다.
- 쌍 채점은 기존 pair 초안의 완료된 packet ID를 재사용한다.
- 모델 ID, 프롬프트 hash, 시나리오 bundle hash, 메모리 index hash 또는 packet batch hash가 달라지면 조용히 섞지 않고 stale 오류로 중단한다.
- 실패한 현재 항목은 오류를 기록하고 프로세스를 실패 종료한다. 재실행하면 실패 항목부터 다시 시도한다.
- 최종 활성 점수 배치는 전체 개수가 정확하고 모든 packet hash가 일치할 때만 원자적으로 교체한다.

## CLI

정식 실행은 하나의 명령으로 제공한다.

```bash
npx tsx server/runAmyHoodEvaluationV6.ts formal-run \
  --candidate-base-url http://127.0.0.1:8080/v1 \
  --embedding-base-url http://127.0.0.1:8081/v1 \
  --judge-base-url http://127.0.0.1:8082/v1 \
  --html docs/reports/2026-07-22-amy-hood-evaluation-v6-formal.html
```

최초 실행은 새 experiment group을 만들고 ID를 checkpoint 파일에 기록한다. 같은 명령을 다시 실행하면 그 ID를 재사용한다. `--group`을 명시하면 지정한 기존 그룹만 재개한다. 기존 `create`, `execute`, `resume`, `judge-local`, `judge-pairs-local`, `report` 명령은 진단용으로 유지한다.

## 테스트 전략

### Happy Path

- 5개 반복의 450개 응답, 450개 개별 점수, 225개 pair 점수를 완성하고 보고서를 생성한다.

### Edge Cases

1. 두 번째 반복부터 시작해도 첫 번째 반복 채점 초안과 충돌하지 않는다.
2. 일부 응답과 일부 채점이 완료된 상태에서 재실행하면 완료 항목을 호출하지 않는다.
3. 다섯 반복의 점수가 역순으로 완료되어도 packet ID 기준으로 정확히 450개를 병합한다.

### Failure Paths

- 모델, 프롬프트, 번들, 검색 인덱스 또는 packet hash가 바뀌면 기존 진행 상태를 보존하고 실패한다.
- 90개 반복 배치 중 하나라도 미완료이면 450개 활성 배치와 최종 보고서를 만들지 않는다.
- 응답 서버, 임베딩 서버 또는 Judge 서버 호출 실패 시 부분 결과를 보존하고 재실행 가능 상태로 종료한다.

## 범위 제외

- 평가 문항, 정답 키, 정책 메모리 내용 및 점수 산식 변경
- Tool Calling 도입
- 병렬 모델 호출
- v5 실행 결과 변경
