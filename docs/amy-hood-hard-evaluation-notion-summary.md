# Amy Hood RAG GraphRAG 평가 정리

## 1. 현재 상태

현재는 정식 성능평가 전 단계입니다.

일반 RAG 파이프라인은 다음 단계까지 검증했습니다.

- 원천 데이터 청크화
- bge-m3 기반 로컬 임베딩
- 2017-2019년 데이터 포함 버전 평가
- 2017-2019년 데이터 제외 버전 평가
- WebUI에서 포함/제외 근거 비교 확인

다만 아직 GraphRAG 결과와 점수 기반 비교를 끝낸 상태는 아닙니다.

현재까지의 평가는 일반 RAG가 누락 데이터 상황에서 어떤 대체 근거를 가져오는지 확인한 1차 검증입니다.

## 2. 평가 질문지

어려운 평가셋은 총 15문항입니다.

파일 위치:

`evaluation/amy_hood_hard_eval_questions_15.json`

질문은 3개 영역으로 나누었습니다.

| 영역 | 문항 수 | 평가 목적 |
|---|---:|---|
| 과거 기억 복원 | 5 | Amy Hood의 과거 재무 판단 기준을 근거 기반으로 복원하는지 확인 |
| 미래 예측 | 5 | GitHub 인수 여부를 과거 데이터가 누락된 조건에서도 Amy Hood식으로 판단하는지 확인 |
| 의사결정 유사도 | 5 | 답변이 Amy Hood의 CFO 의사결정 방식과 유사한지 확인 |

## 3. 질문 난이도를 높인 기준

기존 질문은 파이프라인 검증용에 가까웠습니다.

이번 어려운 평가셋은 다음 기준으로 정제했습니다.

1. 단순 사실 확인 질문을 줄였습니다.
2. 여러 재무 변수가 충돌하는 상황을 넣었습니다.
3. 2017-2019년 누락 구간과 관련된 GitHub 인수 여부 판단 문항을 강화했습니다.
4. 답변이 원문 근거를 잘 사용하는지 확인할 수 있게 했습니다.
5. 근거 없는 수치나 일반론을 말하면 감점되도록 채점 기준을 넣었습니다.

## 4. 어려운 질문 예시

### 예시 1

Microsoft가 GitHub 인수를 검토하고 있다. GitHub는 개발자 생태계와 네트워크 효과는 강하지만 단기 수익성은 낮고 인수 가격 부담이 크다. Amy Hood 페르소나라면 인수 승인 여부를 어떤 기준으로 결정할까?

평가 포인트:

- 단기 수익성만으로 판단하지 않는지
- 개발자 생태계와 네트워크 효과를 장기 플랫폼 가치로 보는지
- 인수 가격 부담과 장기 수익화 가능성을 함께 비교하는지
- 승인, 보류, 거절 중 조건부 결론을 제시하는지

### 예시 2

GitHub 인수 후 Microsoft가 제품을 빠르게 통합하면 비용 시너지는 생기지만 개발자 커뮤니티 신뢰가 훼손될 수 있다. Amy Hood 페르소나라면 이 리스크가 인수 여부에 어떤 영향을 준다고 판단할까?

평가 포인트:

- 비용 시너지보다 커뮤니티 신뢰 훼손 리스크를 중시하는지
- 독립 운영 조건을 인수 승인 조건으로 제시하는지
- 개발자 이탈 가능성을 재무 리스크로 해석하는지
- GitHub 인수 의사결정 구조를 재현하는지

### 예시 3

GitHub 인수 가격이 높아 단기 EPS와 operating margin에 부담이 생길 수 있다. Amy Hood 페르소나라면 어느 정도의 단기 희석을 감수하고, 어떤 회수 논리가 있어야 인수를 승인할까?

평가 포인트:

- 단기 EPS 희석과 장기 전략 가치를 함께 비교하는지
- 회수 논리와 수익화 경로를 요구하는지
- 가격 부담을 무시하지 않는지
- 확인되지 않은 시너지는 단정하지 않는지

## 5. 점수표

점수표 파일:

`evaluation/amy_hood_hard_eval_scorecard.csv`

일반 RAG와 GraphRAG를 같은 문항으로 비교할 수 있게 만들었습니다.

각 문항은 5개 기준으로 1점부터 5점까지 평가합니다.

| 기준 | 설명 |
|---|---|
| 근거 정확도 | 답변이 실제 근거 문서와 맞는지 |
| 과거 기억 복원 | 과거 사건, 맥락, 재무 판단 기준을 잘 복원하는지 |
| 미래 적용력 | 새로운 상황에 Amy Hood식 기준을 적용하는지 |
| 의사결정 유사도 | CFO 관점의 판단 구조가 Amy Hood와 유사한지 |
| 환각 억제 | 근거 없는 수치나 단정을 피하는지 |

각 기준은 1점부터 5점까지 입력합니다.

총점은 25점 만점입니다.

## 6. 1차 검증 결과

기존 15문항 평가 기준으로는 다음 결과를 확인했습니다.

비교 요약 파일:

`evaluation/amy_hood_eval_full_vs_holdout_summary.json`

결과:

- 2017-2019 포함 데이터: 1,843개 문서, 3,009개 청크
- 2017-2019 제외 데이터: 1,448개 문서, 2,377개 청크
- 누락 차이: 395개 문서, 632개 청크
- 15문항 중 3문항에서 top evidence 변경

변경된 문항:

| 문항 | 포함 버전 근거 | 제외 버전 근거 |
|---|---|---|
| memory_segment_guidance | fy2018_q3.json | fy2014_q3.json |
| memory_acquisition_integration | fy2018_q2.json | fy2024_q2.json |
| future_ai_infra_margin_pressure | fy2019_q4.json | microsoft_amy_hood.csv |

해석:

2017-2019년 데이터가 빠졌을 때 일부 질문에서는 해당 기간의 직접 근거가 사라졌습니다.

그 경우 일반 RAG는 다른 연도 어닝콜이나 Amy Hood CSV에서 대체 근거를 가져왔습니다.

따라서 현재 결과는 점수 기반 성능평가라기보다, 누락 데이터 상황에서 일반 RAG가 어떤 대체 근거를 선택하는지 확인한 1차 검증입니다.

## 7. 다음 진행 방법

다음 단계는 GraphRAG 결과가 나온 뒤 진행합니다.

1. 같은 질문지 `evaluation/amy_hood_hard_eval_questions_15.json`를 GraphRAG에 입력합니다.
2. GraphRAG 답변과 근거를 저장합니다.
3. `evaluation/amy_hood_hard_eval_scorecard.csv`에 일반 RAG와 GraphRAG 점수를 입력합니다.
4. 문항별로 어떤 방식이 더 나은지 코멘트를 작성합니다.
5. 평균 점수와 강점, 약점을 비교합니다.

## 8. 공유용 요약 문장

현재 일반 RAG는 파이프라인 검증과 1차 평가를 완료했습니다.

15문항 기준으로 2017-2019년 데이터 포함 버전과 제외 버전을 비교했고, 누락된 기간의 직접 근거가 빠졌을 때 다른 연도 어닝콜이나 Amy Hood CSV로 근거가 대체되는 것을 확인했습니다.

다음 단계에서는 GitHub 인수 판단을 포함한 어려운 평가셋 15문항을 기준으로 일반 RAG와 GraphRAG 답변을 모두 생성한 뒤, 근거 정확도, 과거 기억 복원, 미래 적용력, 의사결정 유사도, 환각 억제 기준으로 점수화할 예정입니다.
